# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project status

v1 (immediate 1:1 predict-then-learn), v2 (adjustable real-frame-interval replay buffer), v3 (deeper multi-scale U-Net-style encoder/decoder + stacked ConvLSTM temporal core + GroupNorm residual blocks + SSIM/blended loss + gradient clipping), v4 (learned optical-flow head that warps the previous frame forward as the decoder's residual base + per-scale skip-connection ConvLSTM recurrence + decoder blend mask fusing warped/generated estimates + motion-weighted loss reweighting + motion-delta magnitude loss + flow-smoothness regularization + LR warmup scheduler), v5 (web/browser client-server refactor: the OpenCV desktop app is replaced by a FastAPI/Uvicorn/WebSocket server + browser frontend, since the training machine may not have its own webcam -- the browser now owns webcam capture and display; `webcam.py` is deleted and `main.py` is split into `server.py`/`training.py`/`frame_codec.py`), and v6 (significant architecture upgrade, deliberately going beyond `idea.md`'s original non-goals: the separate bottleneck `ConvLSTMStack` + independent-per-scale `SkipConvLSTMBank` are replaced by one `HierarchicalSTLSTM` -- a PredRNN-style dual-memory core where a temporal cell state `c` persists per scale across time while a single spatiotemporal memory `m` zigzags across scales within one timestep, giving genuine cross-scale information flow the old two-mechanism design lacked; `FlowHead` gains an acceleration term (predicts flow-velocity alongside flow, enabling constant-acceleration rather than only constant-velocity warp extrapolation, toggled by `--flow-acceleration`); a `WorldModelHead` adds a JEPA-style auxiliary loss predicting a future encoder-feature-space latent from the current bottleneck latent, its lookahead target sourced by peeking (never popping) the existing replay buffer -- so it's only active while the delay slider is >= `--world-model-horizon`; and an online `Discriminator` (spectral-norm PatchGAN, LSGAN loss, its own optimizer, an adversarial-weight warmup ramp) adds a generator loss term for sharper predictions. `NextFramePredictor.forward`'s hidden state and return value are now the `PredictorHidden`/`PredictorOutput` dataclasses, not raw tuples), and v7 (an optional, additive rollout-consistency auxiliary loss via `--rollout-weight`/`--rollout-horizon`/`--rollout-every-n-steps`/`--rollout-decay` -- the core predict-then-learn loop is strictly teacher-forced and never trains the model on its own prior predictions, so low one-step loss was compatible with the model diverging once it had to run several steps ahead of ground truth; this adds a true-BPTT K-step self-feeding chain, seeded from the primary loop's own fresh prediction/hidden state each tick, scored against real future frames peeked -- never popped -- from the replay buffer, gated inactive whenever the delay slider is below `--rollout-horizon` + 1 frames including the default delay of 0; `train_hidden`'s per-step `detach_hidden()` lifecycle and the buffer's core pop-based FIFO consumption are untouched by this mechanism), and v8 (the 6 auxiliary-loss terms from v4/v6/v7 -- flow smoothness, flow-acceleration smoothness, motion-delta magnitude, world-model latent consistency, adversarial, rollout-consistency -- were summed with static hand-tuned CLI weights that nothing balanced against each other's scale, so one term could silently dominate or go negligible; each now gets a learned log-variance parameter via `model.py`'s `UncertaintyWeighter` (Kendall/Gal/Cipolla 2018 homoscedastic uncertainty weighting: `weight = exp(-log_var)`, contribution `weight * loss + log_var`, clamped via `--uncertainty-clamp` against this project's noisy single-frame online updates), a genuine `NextFramePredictor` submodule riding the same optimizer/gradient-clipping/reset treatment as `world_model_head` -- NOT a separate object, so it needed zero changes to optimizer construction or the "Reset" control's weight-preservation convention. The 6 corresponding CLI flags (`--flow-smoothness-weight` etc.) now only seed each term's INITIAL weight rather than fixing it, and a flag of 0 still fully disables that term; the base photometric loss stays an unweighted anchor so a runaway auxiliary weight can never zero out the primary prediction objective; the adversarial term's existing warmup ramp now scales the whole learned-weight term (including its log-variance regularizer), not just the raw GAN loss, to avoid the ramp pathologically pinning that weight to its clamp ceiling during cold-start. `--motion-loss-weight` (a per-pixel reweighting map, not a competing additive term) and `--ssim-weight` (a blend factor within the base loss) are deliberately untouched by this mechanism), and v9 (a cohesive next-generation redesign per user direction, deliberately superseding -- not just extending -- idea.md's original non-goals: rather than the disconnected v6 `WorldModelHead` forecasting one fixed-horizon future latent via its own small network, plus the separate v7 rollout-consistency `compute_rollout_loss` self-feeding chain scored only in pixel space, `training.py`'s `compute_multistep_losses` unifies both into ONE self-feeding chain scored, at every step, both in pixel space (the old rollout term) and in latent space via `NextFramePredictor.encode_frame` against a stop-gradient real-frame embedding (the old world-model term) -- `model.py`'s `WorldModelHead` class is deleted outright and `world_model_consistency_loss` is renamed `latent_consistency_loss` (a `NextFramePredictor.latent_proj` 1x1 conv + `project_latent()` method is the one thing that survives from `WorldModelHead`'s role: `bottleneck_latent` and `encode_frame()`'s output generally live at different channel counts, so this is a pure channel-count adapter with no forward-in-time prediction of its own, not a reintroduction of a separate forecasting network); `--world-model-*`/`--rollout-*` CLI flags become `--multistep-pixel-weight`/`--multistep-latent-weight`/`--multistep-horizon`/`--multistep-every-n-steps`/`--multistep-decay`, and `UncertaintyWeighter`'s term set gains `multistep_pixel`/`multistep_latent` in place of `world_model`/`rollout`. Two new user-facing capabilities ride alongside this unification: the delay slider now DRIVES the prediction pane's look-ahead directly -- when the delay is above 0, `TrainingEngine._run`'s buffered branch calls `self.generate(target_lag)` (a self-feeding, no-grad rollout seeded from -- and never disturbing -- the CURRENT `train_hidden`/`train_pred`) to produce a genuine `target_lag`-frame forecast, displays one forecast frame per incoming real frame via the existing `pred_jpeg`/`stats` output pair (`stats["mode_label"]` becomes `"FORECAST"`, with `forecast_step`/`forecast_horizon` showing progress through the window), and regenerates a fresh forecast using the newest trained weights the moment the current one is exhausted -- this REPLACES both the old cosmetic single-step `display_hidden`/`display_pred` free-run (v5-v8) and an earlier, since-reverted design in this same v9 pass that exposed `generate()` as a separate on-demand "Generate" button/WebSocket message/binary tag; that separate-button design was wrong (the user explicitly wants "delay N frames" to mean both "train N frames behind" and "show N frames ahead" as ONE mechanism, not a manually-triggered add-on) and has been fully removed -- no `request_generate`/`pop_generated_frame`, no `\x02` tag, no `generate_start`/`generate_end`, no `--max-generate-frames` flag, no Generate button. And checkpoint save/load (`TrainingEngine.save_checkpoint`/`.load_checkpoint`, persisting model/optimizer/discriminator WEIGHTS only -- never hidden state, the replay buffer, or frame data, the same convention `_do_reset` already established for Reset -- autosaved every `--checkpoint-interval-seconds` and on-demand via a web "Save Checkpoint" control, loaded automatically at startup unless `--fresh-start`) is unaffected by that reversal and stands as designed. The flow head (with its acceleration term and blend mask), the hierarchical ST-LSTM core, and the discriminator/adversarial loss are otherwise untouched by v9.) are all implemented across `server.py` (FastAPI app/WebSocket entrypoint), `training.py` (the predict-then-learn loop as a background-thread `TrainingEngine`), `frame_codec.py` (JPEG↔tensor conversion), `model.py`, `static/` (browser frontend), and `config.py`. Dependencies are managed via `pyproject.toml`/`uv.lock`, not `requirements.txt`. `idea.md` has been rewritten as of v9 to reflect the current design directly (its Core behavior, Model architecture, Loss function, Configuration, and Non-goals sections now describe the unified multistep mechanism, the delay-driven look-ahead forecast, and checkpoint save/load as first-class, not exceptions) -- treat it as the authoritative, currently-accurate spec, with the v1-v9 history above kept only as the narrative of how each capability was arrived at.

## What this project is

A Python application that trains a neural network **online, in real time**, on a live webcam feed to predict the next video frame, serving a self-hosted web page with a live side-by-side view of `[real frame | model prediction]` while it learns. The browser (not necessarily the machine doing the training) owns the webcam and both display panes; the server just trains and streams predictions back. Full spec: [idea.md](idea.md).

Key behavioral requirements from the spec (do not silently drop these when implementing):

- **Predict-then-learn loop, one step behind**: the model predicts the next frame *before* it arrives; when the ground-truth frame arrives (live, or popped from the replay buffer — see below), loss is computed against the earlier prediction, then backprop/step happens, then that frame is fed in to produce the *next* prediction. This ordering is the non-obvious core of the app — implement and comment it explicitly.
- **Temporal memory is required**: encoder → recurrent temporal core (ConvLSTM or equivalent) → decoder. A single-frame-in/single-frame-out model does not satisfy the spec because it can't represent motion. Recurrence lives at both the bottleneck (`ConvLSTMStack`) *and* each encoder skip-connection scale (`SkipConvLSTMBank`, one independent ConvLSTM cell per scale, toggleable via `--skip-lstm-base-channels 0`) — skip-only-feedforward connections cap how well fast/fine motion detail tracks.
- **Motion is modeled explicitly, not just regressed**: `FlowHead` predicts a per-step optical-flow field (conditioned on its own previous flow estimate, so it refines rather than re-derives each frame) used to warp the current frame/skip activations forward under a constant-velocity assumption; the decoder optionally blends that warped estimate with a freely generated one via a predicted per-pixel mask (`--blend-mask`) so it isn't limited to a small bounded delta in occlusion/disocclusion regions it can't warp correctly.
- **Detach hidden state from autograd after every training step, not just optimizer steps** — backprop must span exactly one timestep, whether that step trained on the live frame or a delayed one popped from the buffer, or the graph grows unbounded.
- **Frame intake happens on a background thread** (the `TrainingEngine`'s own thread) with a lock-protected "latest frame" mailbox (`LatestSlot`), fed by the async WebSocket receive handler, so training/inference is never blocked waiting on the network -- a direct generalization of the old `WebcamStream` pattern with the browser as the frame producer instead of `cap.read()`.
- Internally downsample for training (default ~96×72, must stay easily configurable), upscale only for display.
- Auto-detect GPU, fall back to CPU.
- Loss function should be swappable (MSE baseline, but structure so SSIM/perceptual loss can replace it later). The final training loss composes this swappable base loss with three optional additive/reweighting terms (all wired through `compute_training_loss` in `main.py`): per-pixel motion-weighted reweighting of the base loss (`--motion-loss-weight`, via `motion_weight_map`), a motion-delta magnitude term supervising *how much* the frame changed (`--motion-delta-weight`, via `motion_delta_loss`), and edge-aware flow-smoothness regularization (`--flow-smoothness-weight`, via `flow_smoothness_loss`) — the flow head has no other direct supervision.
- Runs indefinitely; a "Stop" web control pauses training (state preserved, not cleared -- any new inbound frame implicitly resumes it) without restarting the server process, and a "Reset" control resets recurrent hidden state + optimizer state + replay buffer without restarting the process. An actual process shutdown is deliberately not exposed as a web control -- that stays an operator action (Ctrl-C).
- **Real-frame delay is adjustable live** via a browser slider ("Real frame delay (frames)") that snaps to whole-frame detents and shows the current value (also echoed in the stats readout), sent to the server as a `{"type":"set_delay","frames":N}` WebSocket control message: at 0 the model trains on every frame immediately (original 1:1 behavior); above 0, real frames are pushed into a small in-memory FIFO replay buffer and drained one-per-iteration by training, so training lags real time by exactly that many frames but still eventually trains on *every* frame — nothing is discarded. This is `training.py`'s `TrainingEngine._run`'s `train_hidden`/`train_pred`/`_buffer` path. As of v9, this same delay value N ALSO drives what the prediction pane displays (see below) — it is one unified mechanism, not two.
- **The display's prediction pane shows a genuine N-frame look-ahead forecast** when buffering is active (v9; replaces the earlier per-frame cosmetic `display_hidden`/`display_pred` free-run from v5-v8): `TrainingEngine._run`'s buffered branch calls `self.generate(target_lag)` — a `torch.no_grad()` self-feeding rollout seeded from the CURRENT `train_hidden`/`train_pred` — to produce `target_lag` forecast frames, shows one per incoming real frame, and regenerates a fresh forecast (using whatever the model has learned since) the instant the current one runs out. It never produces gradients and never touches the replay buffer. `stats["mode_label"]` is `"FORECAST"` (with `forecast_step`/`forecast_horizon`) whenever this is active, `"REAL"` at delay=0. There is no separate on-demand control for this — see idea.md's Core behavior and Non-goals sections.
- No disk-backed dataset/standalone N-steps-ahead rollout API/config UI beyond the one delay slider above — explicitly out of scope (see idea.md's Non-goals section before adding any of these). The replay buffer itself is in-scope but stays a small bounded in-memory FIFO queue, not a persisted dataset. Model/optimizer/discriminator checkpointing (weights only, see v9 above) is in-scope, not a non-goal. The learned optical-flow head (`FlowHead`) is in-scope, not a non-goal — see idea.md's Model architecture section; only *additional* auxiliary input modalities beyond it (depth, segmentation, audio, etc.) remain out of scope.

## Stack

- Python, PyTorch (model/training), FastAPI + Uvicorn + WebSockets (client/server transport), OpenCV `cv2` (server-side JPEG codec/resize only, no GUI window).
- Dependencies (torch, opencv-python, numpy, fastapi, uvicorn) are managed by `uv` via `pyproject.toml`/`uv.lock`.
- Config values (server host/port, TLS cert/key paths, working resolution, display upscale factor, learning rate + warmup schedule, encoder depth/channel counts, ConvLSTM layer count/hidden channels, per-scale skip-recurrence channels, flow-head channels/smoothness weight, blend-mask toggle, motion-loss/motion-delta weights, optimizer, loss choice/blend weight, gradient-clip norm, real-frame interval) should be exposed as easily editable constants or CLI flags, not buried in logic.

## Commands

This is a `uv` project (`pyproject.toml` + `uv.lock`).

```
uv sync                                          # install/sync dependencies into .venv
uv run server.py                                 # run with defaults (96x72, upscale 6x, port 8000)
uv run server.py --help                          # see all CLI flags (resolution, lr, host/port, etc.)
uv add <package>                                  # add a new dependency (updates pyproject.toml + uv.lock)

# One-time: generate a self-signed TLS cert (browsers require a secure
# context -- https:// or localhost -- for getUserMedia camera access):
openssl req -x509 -newkey rsa:2048 -nodes -keyout key.pem -out cert.pem -days 365 -subj "/CN=krystalball"
uv run server.py --ssl-keyfile key.pem --ssl-certfile cert.pem

# Then open https://<this-machine's-lan-ip>:8000/ in a browser on any device
# on the same network and grant camera permission (click through the
# self-signed cert warning -- expected).
```

No lint/test tooling exists yet.

## Code layout

- `config.py` — argparse CLI flags / defaults (server host/port, TLS cert/key paths, resolution, lr + warmup, encoder scales/channels/res-blocks, ConvLSTM layers/hidden channels, skip-recurrence channels, flow-head/blend-mask/motion-loss flags, optimizer, loss choice + SSIM blend weight, gradient-clip norm).
- `model.py` — multi-scale U-Net-style `Encoder` / `Decoder` (GroupNorm `ResBlock`s via `DownBlock`/`UpBlock`, one skip connection per scale) → `HierarchicalSTLSTM` (a stack of `STLSTMCell`s spanning skip scales and bottleneck depth, PredRNN-style dual memory: per-scale temporal `c` + one shared cross-scale spatiotemporal `m` that zigzags finest-to-coarsest each timestep; hidden state is `STLSTMHiddenState(hc: list[(h, c)], m: Tensor)`) → `NextFramePredictor`, which also owns `FlowHead`/`warp_frame` (optical-flow + velocity prediction and forward-warp), the decoder's blend-mask fusion, a stateless `encode_frame()` history-free encoder pass consumed by the v9 unified multistep loss (see `training.py`), and a `latent_proj`/`project_latent()` 1x1 conv (v9) that adapts `bottleneck_latent`'s channel count to `encode_frame()`'s so `latent_consistency_loss` can compare them -- a pure channel adapter, not a forecasting network. Plus `detach_hidden()` (generic recursive detach over the hidden-state dataclass/list/tuple structure), the auxiliary-loss helpers (`motion_weight_map`, `motion_delta_loss`, `flow_smoothness_loss`, `latent_consistency_loss` -- renamed from `world_model_consistency_loss` in v9, now scored at every step of the unified multistep chain instead of one fixed horizon via a separate `WorldModelHead`, which v9 deletes outright -- `lsgan_d_loss`/`lsgan_g_loss`), `UncertaintyWeighter` (the learned per-term loss-weighting submodule, `NextFramePredictor.loss_weighter` — see v8), `Discriminator` (spectral-norm PatchGAN, lives outside `NextFramePredictor`), and the `get_loss_fn()` swappable-loss factory (`mse` / `ssim` / `mse_ssim`, the last via `SSIMLoss`/`BlendedLoss`).
- `frame_codec.py` — `decode_jpeg_to_tensor()`/`tensor_to_jpeg()`: JPEG bytes ↔ model tensor conversion for the WebSocket transport (replaces the old `preprocess()`/`tensor_to_bgr()`, which worked directly on `cv2.VideoCapture` frames/an `imshow` canvas).
- `training.py` — `TrainingEngine`: owns the model/optimizer/discriminator/hidden-state and runs the predict-then-learn loop (including `compute_training_loss`, which composes the base loss with the auxiliary terms above -- see v9's `compute_multistep_losses`, which replaces the old separate `compute_rollout_loss`/`WorldModelHead` stash with one self-feeding chain scored both in pixel space and, via `encode_frame`, latent space -- and the Adam/SGD + `LinearLR` warmup scheduler built by `make_optimizer_and_scheduler`) on its own background thread for the process lifetime. `LatestSlot` is the lock-protected single-item mailbox used for both frame intake (fed by the async WebSocket handler) and prediction/stats output. v9 also adds `save_checkpoint()`/`load_checkpoint()`, persisting/restoring model+optimizer+discriminator weights only (never hidden state/buffer/frame data, same convention as `_do_reset`), autosaved on an interval and loaded automatically at startup unless `--fresh-start`; and `generate()`, a no-grad self-feeding rollout seeded from `train_hidden`/`train_pred` without disturbing them, called internally by `_run`'s buffered branch (`self.generate(target_lag)`) to (re)generate the `self._display_rollout` list that becomes the delay-driven look-ahead forecast shown in the prediction pane -- NOT exposed as a separate on-demand control.
- `server.py` — FastAPI app/entrypoint: creates the one `TrainingEngine` in the app lifespan, mounts `static/` for the frontend, and exposes the `/ws` WebSocket endpoint (binary JPEG frames both directions, one tag (`\x01`) covering both the live one-step prediction and the v9 delay-driven forecast -- `stats["mode_label"]` distinguishes REAL from FORECAST -- and JSON for config/stats/control messages: `set_delay`/`reset`/`stop` plus v9's `save_checkpoint`).
- `static/` — browser frontend (`index.html`/`app.js`/`style.css`): `getUserMedia` webcam capture, WebSocket client, canvas rendering of both panes (the real pane draws directly from the local camera, no round trip needed), delay slider + Reset/Stop buttons, and (v9) a Save Checkpoint button. The stats line shows `FORECAST step/horizon` instead of the old `FREE-RUN` label whenever the delay is above 0.
