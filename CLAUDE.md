Two additive, non-flag-gated stabilization mechanisms close this gap: each `BottleneckAttentionBlock` gains `attn_gate`/`ffn_gate`, a pair of learnable scalars (one per residual branch) clamped to `[0, 1]` at use -- the same convention as `Decoder.mask_gate` -- giving the optimizer a persistent, cheap lever to throttle a misbehaving branch back toward inert for the rest of training, which zero-init alone never provided; and `BottleneckAttentionStack` gains `out_norm`, a `GroupNorm` applied not to its raw output (which would rescale the real bottleneck latent even at construction, since `GroupNorm(x0) != x0` in general) but to its cumulative *delta* (`x_final - x0`) before that's added back, bounding the aggregate contribution reaching the decoder without disturbing the "identity at construction" guarantee -- verified bit-exact (`bottleneck_latent`/`pred` identical to `--attn-heads 0` at construction, same seed). Both mechanisms zero-init to a true no-op and compose cleanly with each other and with v14's existing zero-init out_proj/ffn2. Enabling this, like v14 itself, breaks loading an on-disk checkpoint saved under prior code (new `attn_gate`/`ffn_gate`/`out_norm.*` state_dict keys, `strict=True`) -- `--fresh-start`/Delete-Checkpoint is the expected recovery.) are all implemented across `server.py` (FastAPI app/WebSocket entrypoint), `training.py` (the predict-then-learn loop as a background-thread `TrainingEngine`), `frame_codec.py` (JPEG↔tensor conversion), `model.py`, `static/` (browser frontend), and `config.py`. Dependencies are managed via `pyproject.toml`/`uv.lock`, not `requirements.txt`. `idea.md` has been rewritten as of v9 (and updated again for v13's learned upsample head and v14's bottleneck attention block) to reflect the current design directly (its Core behavior, Model architecture, Loss function, Configuration, and Non-goals sections now describe the unified multistep mechanism, the delay-driven look-ahead forecast, checkpoint save/load, the learned display-resolution upsample head, and the global bottleneck self-attention block as first-class, not exceptions) -- treat it as the authoritative, currently-accurate spec, with the v1-v14 history above kept only as the narrative of how each capability was arrived at.

## What this project is

A Python application that trains a neural network **online, in real time**, on a live webcam feed to predict the next video frame, serving a self-hosted web page with a live side-by-side view of `[real frame | model prediction]` while it learns. The browser (not necessarily the machine doing the training) owns the webcam and both display panes; the server just trains and streams predictions back. Full spec: [idea.md](idea.md).

Key behavioral requirements from the spec (do not silently drop these when implementing):

- **Predict-then-learn loop, one step behind**: the model predicts the next frame *before* it arrives; when the ground-truth frame arrives (live, or popped from the replay buffer — see below), loss is computed against the earlier prediction, then backprop/step happens, then that frame is fed in to produce the *next* prediction. This ordering is the non-obvious core of the app — implement and comment it explicitly.
- **Temporal memory is required**: encoder → recurrent temporal core (ConvLSTM or equivalent) → decoder. A single-frame-in/single-frame-out model does not satisfy the spec because it can't represent motion. Recurrence lives at both the bottleneck (`ConvLSTMStack`) *and* each encoder skip-connection scale (`SkipConvLSTMBank`, one independent ConvLSTM cell per scale, toggleable via `--skip-lstm-base-channels 0`) — skip-only-feedforward connections cap how well fast/fine motion detail tracks.
- **Motion is modeled explicitly, not just regressed**: `FlowHead` predicts a per-step optical-flow field (conditioned on its own previous flow estimate, so it refines rather than re-derives each frame) used to warp the current frame/skip activations forward under a constant-velocity assumption; the decoder optionally blends that warped estimate with a freely generated one via a predicted per-pixel mask (`--blend-mask`) so it isn't limited to a small bounded delta in occlusion/disocclusion regions it can't warp correctly.
- **Detach hidden state from autograd after every training step, not just optimizer steps** — backprop must span exactly one timestep, whether that step trained on the live frame or a delayed one popped from the buffer, or the graph grows unbounded.
- **Loss spike guard** (v10): before every `.backward()`, `TrainingEngine._is_loss_spike()` rejects the optimizer/discriminator step for that tick — weights untouched — if the composed loss is non-finite (always) or a rolling-outlier (`--loss-spike-guard`, default on; threshold via `--loss-spike-threshold` std-multiplier over `--loss-spike-min-history`+ samples). The forward pass and `detach_hidden()` still happen normally on a rejected tick, so prediction/display never stall and backprop still spans exactly one timestep either way. A rejected tick's own loss value must NOT be appended to `_loss_history`/`_loss_component_history` — it would poison the rolling baseline used to catch the next spike and skew the on-screen average toward loss nothing was trained on.
- **Loss spike guard auto-reset** (v12): a SUSTAINED high rejection rate (at least `--loss-spike-auto-reset-threshold` rejections in the last `--loss-spike-auto-reset-window` ticks, tracked by `_loss_spike_window`) auto-triggers the SAME full reset the manual Reset button performs (`TrainingEngine.request_reset()` — hidden state, replay buffer, and optimizer/scheduler state all cleared/reinitialized; model/discriminator WEIGHTS untouched), not a bespoke partial reset. `_loss_spike_window`/`_loss_spike_count` are cleared as part of that reset (`_do_reset`), so it takes a fresh run of rejections to fire again, not every subsequent tick.
- **Frame intake happens on a background thread** (the `TrainingEngine`'s own thread) with a lock-protected "latest frame" mailbox (`LatestSlot`), fed by the async WebSocket receive handler, so training/inference is never blocked waiting on the network -- a direct generalization of the old `WebcamStream` pattern with the browser as the frame producer instead of `cap.read()`.
- Internally downsample for training (default ~96×72, must stay easily configurable). Display-resolution output is now (v13) produced by a genuinely **learned** upsample head (`model.py`'s `SuperResHead`, trained online via a `superres` loss term), not a fixed bicubic/nearest-neighbor stretch -- see v13 in Project status above and idea.md's Resolution/Model architecture sections.
- Auto-detect GPU, fall back to CPU.
- Loss function should be swappable (MSE baseline, but structure so SSIM/perceptual loss can replace it later). The final training loss composes this swappable base loss with three optional additive/reweighting terms (all wired through `compute_training_loss` in `main.py`): per-pixel motion-weighted reweighting of the base loss (`--motion-loss-weight`, via `motion_weight_map`), a motion-delta magnitude term supervising *how much* the frame changed (`--motion-delta-weight`, via `motion_delta_loss`), and edge-aware flow-smoothness regularization (`--flow-smoothness-weight`, via `flow_smoothness_loss`) — the flow head has no other direct supervision.
- Runs indefinitely; a "Stop" web control pauses training (state preserved, not cleared -- any new inbound frame implicitly resumes it) without restarting the server process, and a "Reset" control resets recurrent hidden state + optimizer state + replay buffer without restarting the process. An actual process shutdown is deliberately not exposed as a web control -- that stays an operator action (Ctrl-C).
- **Real-frame delay is adjustable live** via a browser slider ("Real frame delay (frames)") that snaps to whole-frame detents and shows the current value (also echoed in the stats readout), sent to the server as a `{"type":"set_delay","frames":N}` WebSocket control message: at 0 the model trains on every frame immediately (original 1:1 behavior); above 0, real frames are pushed into a small in-memory FIFO replay buffer and drained one-per-iteration by training, so training lags real time by exactly that many frames but still eventually trains on *every* frame — nothing is discarded. This is `training.py`'s `TrainingEngine._run`'s `train_hidden`/`train_pred`/`_buffer` path. As of v9, this same delay value N ALSO drives what the prediction pane displays (see below) — it is one unified mechanism, not two.
- **The display's prediction pane shows a genuine N-frame look-ahead forecast** when buffering is active (v9; replaces the earlier per-frame cosmetic `display_hidden`/`display_pred` free-run from v5-v8): `TrainingEngine._run`'s buffered branch calls `self.generate(target_lag)` — a `torch.no_grad()` self-feeding rollout seeded from the CURRENT `train_hidden`/`train_pred` — to produce `target_lag` forecast frames, shows one per incoming real frame, and regenerates a fresh forecast (using whatever the model has learned since) the instant the current one runs out. It never produces gradients and never touches the replay buffer. `stats["mode_label"]` is `"FORECAST"` (with `forecast_step`/`forecast_horizon`) whenever this is active, `"REAL"` at delay=0. There is no separate on-demand control for this — see idea.md's Core behavior and Non-goals sections.
- No disk-backed dataset/standalone N-steps-ahead rollout API/config UI beyond the one delay slider above — explicitly out of scope (see idea.md's Non-goals section before adding any of these). The replay buffer itself is in-scope but stays a small bounded in-memory FIFO queue, not a persisted dataset. Model/optimizer/discriminator checkpointing (weights only, see v9 above) is in-scope, not a non-goal. The learned optical-flow head (`FlowHead`) is in-scope, not a non-goal — see idea.md's Model architecture section; only *additional* auxiliary input modalities beyond it (depth, segmentation, audio, etc.) remain out of scope.

## Stack

- Python, PyTorch (model/training), FastAPI + Uvicorn + WebSockets (client/server transport), OpenCV `cv2` (server-side JPEG codec/resize only, no GUI window).
- Dependencies (torch, opencv-python, numpy, fastapi, uvicorn) are managed by `uv` via `pyproject.toml`/`uv.lock`.
- Config values (server host/port, TLS cert/key paths, working resolution, display upscale factor (also the learned upsample head's fixed scale factor, v13), learning rate + warmup schedule, encoder depth/channel counts, ConvLSTM layer count/hidden channels, per-scale skip-recurrence channels, flow-head channels/smoothness weight, blend-mask toggle, motion-loss/motion-delta weights, upsample-head channels/blocks/delta-scale/loss-weight (v13), optimizer, loss choice/blend weight, gradient-clip norm, loss-spike guard toggle/threshold/min-history, real-frame interval) should be exposed as easily editable constants or CLI flags, not buried in logic.

## Commands

This is a `uv` project (`pyproject.toml` + `uv.lock`).

```
uv sync                                          # install/sync dependencies into .venv
uv run server.py                                 # run with defaults (96x72, upscale 6x, port 8000)
uv run server.py --help                          # see all CLI flags (resolution, lr, host/port, etc.)
uv add <package>                                  # add a new dependency (updates pyproject.toml + uv.lock)

# One-time: generate a self-signed TLS cert (browsers require a secure
# context -- https:// or localhost -- for getUserMedia camera access):
openssl req -x509 -newkey rsa:2048 -nodes -keyout key.pem -out cert.pem -days 365 -subj "/CN=krystalball"
uv run server.py --ssl-keyfile key.pem --ssl-certfile cert.pem

# Then open https://<this-machine's-lan-ip>:8000/ in a browser on any device
# on the same network and grant camera permission (click through the
# self-signed cert warning -- expected).
```

## Code layout

- `config.py` — argparse CLI flags / defaults (server host/port, TLS cert/key paths, resolution, lr + warmup, encoder scales/channels/res-blocks, ConvLSTM layers/hidden channels, skip-recurrence channels, flow-head/blend-mask/motion-loss flags, optimizer, loss choice + SSIM blend weight, gradient-clip norm, v10's `--loss-spike-guard`/`--loss-spike-threshold`/`--loss-spike-min-history`, v12's `--loss-spike-auto-reset-window`/`--loss-spike-auto-reset-threshold`, v13's `--superres-channels`/`--superres-blocks`/`--superres-delta-scale`/`--superres-weight`, v14's `--attn-heads`/`--attn-layers`).
- `model.py` — multi-scale U-Net-style `Encoder` / `Decoder` (GroupNorm `ResBlock`s via `DownBlock`/`UpBlock`, one skip connection per scale) → `HierarchicalSTLSTM` (a stack of `STLSTMCell`s spanning skip scales and bottleneck depth, PredRNN-style dual memory: per-scale temporal `c` + one shared cross-scale spatiotemporal `m` that zigzags finest-to-coarsest each timestep; hidden state is `STLSTMHiddenState(hc: list[(h, c)], m: Tensor)`) → `NextFramePredictor`, which also owns `FlowHead`/`warp_frame` (optical-flow + velocity prediction and forward-warp), the decoder's blend-mask fusion, a stateless `encode_frame()` history-free encoder pass consumed by the v9 unified multistep loss (see `training.py`), a `latent_proj`/`project_latent()` 1x1 conv (v9) that adapts `bottleneck_latent`'s channel count to `encode_frame()`'s so `latent_consistency_loss` can compare them -- a pure channel adapter, not a forecasting network -- and (v13) `SuperResHead`/`self.upsample_head`, a stateless subpixel-conv (`nn.PixelShuffle`) upsampler run on every `forward()` call to produce `PredictorOutput.pred_sr` at display resolution, zero-initialized so it starts pixel-identical to a plain bilinear upsample (`--superres-channels 0` or `--upscale <= 1` disables it, falling back to that same bilinear op so `pred_sr` is always populated). Plus `detach_hidden()` (generic recursive detach over the hidden-state dataclass/list/tuple structure), the auxiliary-loss helpers (`motion_weight_map`, `motion_delta_loss`, `flow_smoothness_loss`, `latent_consistency_loss` -- renamed from `world_model_consistency_loss` in v9, now scored at every step of the unified multistep chain instead of one fixed horizon via a separate `WorldModelHead`, which v9 deletes outright -- `lsgan_d_loss`/`lsgan_g_loss`), `UncertaintyWeighter` (the learned per-term loss-weighting submodule, `NextFramePredictor.loss_weighter` — see v8, now also carrying v13's `superres` term), `Discriminator` (spectral-norm PatchGAN, lives outside `NextFramePredictor`, still scored only against the internal-resolution `pred`, not `pred_sr` -- v13 deliberately doesn't extend adversarial supervision to the upsampled output), and the `get_loss_fn()` swappable-loss factory (`mse` / `ssim` / `mse_ssim`, the last via `SSIMLoss`/`BlendedLoss`, also reused by v13's `superres` reconstruction term). v14 adds `BottleneckAttentionBlock`/`BottleneckAttentionStack` (GroupNorm pre-norm `nn.MultiheadAttention` + conv-space FFN, zero-initialized so identity at construction, plus a learned positional embedding since attention -- unlike every conv in this file -- is permutation-invariant over spatial positions), applied to `bottleneck_latent` inside `NextFramePredictor.forward` right after the ST-LSTM stack produces it and before the decoder consumes it; `--attn-heads 0` (or `--attn-layers 0`) leaves `self.bottleneck_attn` as `None` and skips the block outright, no dataclass or hidden-state changes involved. v15 adds `attn_gate`/`ffn_gate` (per-block, clamped-`[0,1]` learnable scalars, same convention as `Decoder.mask_gate`) and `BottleneckAttentionStack.out_norm` (a `GroupNorm` applied to the stack's cumulative delta, not its raw output) to fix chroma-noise from the block's contribution growing unboundedly past its zero-init identity once training proceeds -- see v15 in Project status above.
- `frame_codec.py` — JPEG bytes ↔ model tensor conversion for the WebSocket transport. v13 splits the old single `decode_jpeg_to_tensor()` into `decode_jpeg_to_bgr()` (imdecode only) + `bgr_to_tensor()` (resize-if-needed + normalize) so one inbound JPEG can be resized to two different resolutions (internal working res for the model input, display res for `SuperResHead`'s ground truth) without paying for `cv2.imdecode` twice per tick; `decode_jpeg_to_tensor()` survives as a thin wrapper of the two for single-resolution callers. `tensor_to_jpeg()` is unchanged (already resolution-agnostic). Originally replaced the pre-v5 `preprocess()`/`tensor_to_bgr()`, which worked directly on `cv2.VideoCapture` frames/an `imshow` canvas.
- `training.py` — `TrainingEngine`: owns the model/optimizer/discriminator/hidden-state and runs the predict-then-learn loop (including `compute_training_loss`, which composes the base loss with the auxiliary terms above -- see v9's `compute_multistep_losses`, which replaces the old separate `compute_rollout_loss`/`WorldModelHead` stash with one self-feeding chain scored both in pixel space and, via `encode_frame`, latent space -- and the Adam/SGD + `LinearLR` warmup scheduler built by `make_optimizer_and_scheduler`) on its own background thread for the process lifetime. `LatestSlot` is the lock-protected single-item mailbox used for both frame intake (fed by the async WebSocket handler) and prediction/stats output. v9 also adds `save_checkpoint()`/`load_checkpoint()`, persisting/restoring model+optimizer+discriminator weights only (never hidden state/buffer/frame data, same convention as `_do_reset`), autosaved on an interval and loaded automatically at startup unless `--fresh-start`; and `generate()`, a no-grad self-feeding rollout seeded from `train_hidden`/`train_pred` without disturbing them, called internally by `_run`'s buffered branch (`self.generate(target_lag)`) to (re)generate the `self._display_rollout` list that becomes the delay-driven look-ahead forecast shown in the prediction pane (v13: now display-resolution `pred_sr` frames, while the self-feeding chain itself still recurs at internal resolution) -- NOT exposed as a separate on-demand control. v10 adds `_is_loss_spike()`/`_loss_spike_baseline()`, called from `_predict_and_learn` before every `.backward()` to reject a non-finite or rolling-outlier tick's optimizer/discriminator step (see Project status above); `_loss_spike_count` is reset in `_do_reset` and surfaced via `stats["loss_spike_count"]`. v11 adds `_build_model()`/`_build_discriminator()` (factored out of `__init__` for reuse) and `_do_delete_checkpoint_and_restart()`/`request_delete_checkpoint_and_restart()`, which delete the on-disk checkpoint and reinitialize the model/discriminator from scratch in place -- see Project status above. v12 adds `_reset_optimizer_state()` (the optimizer-only subset of `_do_reset`, factored out so `_do_reset` itself calls it) and `_loss_spike_window`, which `_predict_and_learn` uses to call `self.request_reset()` -- the same request the manual Reset button sends -- when the recent loss-spike-guard rejection rate crosses `--loss-spike-auto-reset-threshold` -- see Project status above. v13 adds the `BufferedFrame(internal, target_sr)` dataclass (replacing bare tensors in `self._buffer` so a delayed pop still yields both resolutions of the same corresponding frame) and `self.target_width`/`self.target_height` (`= args.width/height * args.upscale`); `_predict_and_learn`/`compute_training_loss` gain `ground_truth_sr`/`pred_sr`/`target_sr` parameters threaded alongside the existing internal-resolution ones. v14's `_build_model()` passes `attn_heads`/`attn_layers`/`height`/`width` through to `NextFramePredictor(...)` -- the latter two solely so the attention block's positional embedding can be sized at construction time (before `make_optimizer_and_scheduler` reads `self.model.parameters()`), otherwise unused by the rest of `training.py`.
- `server.py` — FastAPI app/entrypoint: creates the one `TrainingEngine` in the app lifespan, mounts `static/` for the frontend, and exposes the `/ws` WebSocket endpoint (binary JPEG frames both directions, one tag (`\x01`) covering both the live one-step prediction and the v9 delay-driven forecast -- `stats["mode_label"]` distinguishes REAL from FORECAST -- and JSON for config/stats/control messages: `set_delay`/`reset`/`stop` plus v9's `save_checkpoint` and v11's `delete_checkpoint`). Unchanged by v13 -- the existing `width`/`height`/`upscale` config fields already let the client derive display resolution itself.
- `static/` — browser frontend (`index.html`/`app.js`/`style.css`): `getUserMedia` webcam capture, WebSocket client, canvas rendering of both panes (the real pane draws directly from the local camera, no round trip needed), delay slider + Reset/Stop buttons, (v9) a Save Checkpoint button, and (v11) a red "Delete Checkpoint & Start Over" button gated behind a browser `confirm()` since it's the one irreversible control. The stats line shows `FORECAST step/horizon` instead of the old `FREE-RUN` label whenever the delay is above 0. v13: `sendCanvas` now captures/sends frames at display resolution (`width*upscale x height*upscale`) instead of internal resolution, since the server needs real high-res pixels to train `SuperResHead` against; `predCanvas`'s backing buffer is now display resolution 1:1 (the old CSS `style.width/height` stretch and `style.css`'s `image-rendering: pixelated` rule are removed -- the server now sends an already-correctly-sized, smoothly-upsampled JPEG).
