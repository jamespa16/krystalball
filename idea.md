# Project: Live Webcam Next-Frame Prediction

## Goal
Build a Python application that trains a neural network **online, in real time**, on a live webcam feed to predict the next video frame -- and, when the delay slider is raised, continuously shows a genuine look-ahead forecast of what happens that many frames into the future. While training, serve a self-hosted web page showing the current webcam frame side-by-side with the model's current prediction, updating live so the user can watch the model learn -- the browser (not necessarily the machine doing the training) owns the webcam.

## Core behavior

1. Continuously receive frames captured by the browser client's webcam, streamed to the server over a WebSocket connection (the server itself has no camera of its own -- see Deployment / Architecture below).
2. Maintain a model with temporal memory (it must condition on frame history, not just the current frame in isolation — a single-frame-in/single-frame-out model has no way to represent motion).
3. Online training loop:
   - The model has already produced a prediction for "the next frame" based on frames seen so far.
   - When the next ground-truth frame is available (see the replay buffer below — this may be the live frame, or a slightly delayed one), compute a loss between the pending prediction and that frame, backprop, and step the optimizer.
   - Feed that same ground-truth frame into the model to update its temporal state and produce a prediction for the *following* frame.
   - **Adjustable real-frame delay (replay buffer), unified with the display forecast:** a live on-screen slider, snapping to whole-frame detents and displaying the current value, sets how many frames N training is allowed to lag behind live video, from 0 (train on every frame immediately, the original 1:1 behavior) up to a configurable max (default 60 frames). Above 0, incoming real frames are appended to a small in-memory FIFO replay buffer instead of being trained on right away; a separate consumption step drains at most one frame per loop iteration once the backlog exceeds the target delay. This means training lags real time by exactly the slider's frame count, but it still eventually trains on *every* real frame — nothing is discarded, unlike a pure free-run/self-feed scheme. (If the slider is lowered mid-run, the buffer drains faster than it fills — more than one frame per iteration — until the backlog catches back down to the new, smaller target.) "Delay N frames" is ONE mechanism, not two: it also drives what the prediction pane shows (see below), not just how far training lags.
   - **Delay-driven look-ahead forecast, not a separate control:** whenever the delay is above 0, the prediction pane shows a genuine N-frame forecast -- generated by self-feeding the model forward from its CURRENT (latest-trained) live state under no_grad, consuming one forecast frame per incoming real frame. Once that N-frame forecast is exhausted, a fresh one is generated immediately using whatever the model has learned in the meantime, so the pane always reflects the newest weights. This never receives gradients and never disturbs the real training state (`train_hidden`/`train_pred`); there is no separate button or on-demand trigger for it -- raising the delay slider is the only control needed.
   - Backprop must still span exactly one timestep: hidden state is detached after every training step (whether that step consumed the live frame directly or a delayed frame popped from the buffer), so an extended lag never grows the autograd graph.
   - **Loss spike guard:** before backprop, that step's fully-composed training loss is checked against the recent rolling loss distribution (and always checked for NaN/Inf, regardless of the toggle below). A non-finite loss, or one that exceeds the rolling mean by more than a configurable multiple of the rolling standard deviation, is rejected: the optimizer/discriminator step for that tick is skipped entirely (weights untouched), while the forward pass and hidden-state detach still happen normally so prediction/display never stall. This exists because online, single-frame-batch training has no minibatch to average a bad frame away and no epoch boundary to recover at -- one corrupted gradient (camera glitch, hand over the lens, a flash) can otherwise permanently drag the model into a bad region, which previously only the manual Reset control could recover from.
   - **Loss spike guard auto-reset:** an isolated rejection is expected and left alone, but a SUSTAINED high rejection rate over a rolling window of recent ticks (both thresholds configurable) means something the per-tick guard alone can't fix -- so it automatically triggers the SAME reset the manual Reset control performs (recurrent hidden state, the replay buffer, and optimizer/scheduler state all cleared/reinitialized; model/discriminator weights untouched), rather than any smaller bespoke reset of its own. The rejection-rate window is cleared as part of that reset, so it takes a fresh run of rejections to trigger again rather than re-firing every subsequent tick.
4. Render two panes side by side in a browser page: `[current real frame | current prediction]`, updated live. Show basic live stats: frame count, FPS, a rolling-average training loss, and (when the delay is above 0) how far into the current forecast window the displayed frame is.
5. Run indefinitely until stopped. A "Stop" control pauses training (state preserved, not cleared) without restarting the server process -- sending a new frame (e.g. reloading the page and re-granting camera access) implicitly resumes it. A "Reset" control clears the model's temporal hidden state and optimizer state without restarting the process, for recovering from instability -- it does not touch the on-disk checkpoint and PRESERVES learned weights (see below). Actually shutting down the server process remains an operator action (Ctrl-C on the machine running it), not exposed as a web control -- there's no "restart the process" button, deliberately, to avoid a stray browser click taking down shared server-side state. Model/optimizer/discriminator WEIGHTS (never hidden state, the replay buffer, or any frame data) are periodically autosaved to a checkpoint file and reloaded automatically at startup if present, so a server restart resumes already-trained rather than from scratch; a `--fresh-start` flag skips loading an existing checkpoint, and a web "Save Checkpoint" control triggers an on-demand save. A "Delete Checkpoint & Start Over" control is the live equivalent of `--fresh-start`, requiring no server restart: unlike Reset, it discards learned weights entirely -- deleting the on-disk checkpoint file (if any) and reinitializing the model/discriminator from scratch in place -- so the browser confirms before sending it, since it's the one irreversible control on the page.

## Technical requirements

- **Language/stack:** Python/PyTorch for the model (server-side, GPU-capable machine). FastAPI + Uvicorn + WebSockets for the client/server transport. A browser (`getUserMedia` + `<canvas>` + WebSocket) is the client: it captures the webcam and renders both display panes. OpenCV (`cv2`) is used server-side only for JPEG decode/encode and resizing -- no GUI window.
- **Resolution:** Internally downsample webcam frames to a small working resolution (start around 96×72, must be easily configurable) so that training keeps pace with the live video framerate. A learned upsample head brings the prediction back up to display resolution (working resolution x a configurable factor) -- trained online alongside the rest of the model, not a fixed bicubic/nearest-neighbor stretch -- so the browser now also sends a display-resolution frame for that head's ground truth, in addition to (derived from) the internal working-resolution one used as the model's own input.
- **Model architecture:** A multi-scale encoder → a hierarchical spatiotemporal-LSTM recurrent core (PredRNN-style: a temporal cell state persists per scale across time, while a single spatiotemporal memory zigzags across scales within one timestep, giving genuine cross-scale information flow) spanning both the bottleneck and each encoder skip-connection scale → a multi-scale decoder that outputs a predicted RGB frame, optionally blending a flow-warped estimate with a freely-generated one via a learned per-pixel mask. A learned optical-flow head (optionally with an acceleration/velocity term, enabling constant-acceleration rather than only constant-velocity extrapolation) warps the current frame and skip activations forward to form the decoder's residual base, fed its own previous flow (and velocity) estimate each step so it refines rather than re-derives its estimate from scratch. An online discriminator (trained alongside the predictor, adversarial LSGAN loss) pushes predictions to be sharper than photometric loss alone rewards. A **learned upsample head** takes that internal-resolution prediction up to display resolution via subpixel convolution (a residual detail correction, zero-initialized so it starts pixel-identical to a plain bilinear upsample and only improves from training, added on top of that bilinear base) -- stateless, since the recurrent core already owns all temporal modeling; this is the one part of the pipeline resolution-agnostic wrt the internal working resolution, letting a low compute budget stay at a small internal grid while still training genuine display-resolution reconstruction. A **unified multistep self-consistency mechanism** self-feeds the model forward several steps (true backprop-through-time, gradient flowing across the whole chain) and, at every step of that same chain, scores it two ways against a real future frame peeked from the replay buffer: pixel-space reconstruction, and cosine-similarity consistency between the model's own recurrent bottleneck latent and that frame's encoder embedding -- one mechanism producing both a "does the eventual rendered rollout stay accurate" signal and a "does the internal state represent a coherent trajectory" signal, rather than two disconnected forecasting mechanisms. Every auxiliary loss term (flow smoothness, flow-acceleration smoothness, motion-delta magnitude, the two multistep terms, adversarial, the upsample head's display-resolution reconstruction loss) is combined with the base photometric loss via learned homoscedastic uncertainty weighting (each term gets a trained log-variance parameter so its effective weight adapts to its own loss scale, rather than staying at a fixed hand-tuned value) -- the base photometric loss itself is never weighted this way, so it always remains an unweighted anchor no runaway auxiliary term can zero out.
- **Device:** Auto-detect and use GPU if available, otherwise fall back to CPU.
- **Threading:** Training runs in a dedicated background thread with a small lock-protected "latest frame" mailbox (`LatestSlot`) fed by the async WebSocket receive handler, so training/inference is never blocked waiting on the network -- a direct generalization of the original "background thread + lock-protected latest-frame buffer" design, just with the browser as the frame producer instead of `cap.read()`.
- **Loss function:** MSE/SSIM/blended MSE+SSIM between predicted and actual frame is the swappable base loss (structure the code so it's easy to swap for another later). The final training loss composes this base loss with several optional additive auxiliary terms -- edge-aware flow-smoothness regularization (and, if flow-acceleration is enabled, the same regularization on the velocity field), a motion-magnitude term supervising the predicted *amount* of frame-to-frame change, the two halves of the unified multistep self-consistency mechanism (pixel + latent, see Model architecture), the adversarial term, and the upsample head's reconstruction loss (the same swappable base loss function, applied between the head's display-resolution output and a genuine display-resolution ground-truth frame) -- each weighted via learned uncertainty weighting rather than a fixed hand-tuned constant (see Model architecture). A separate per-pixel motion-weighted reweighting of the base loss itself (not a new additive term) upweights regions that actually changed frame-to-frame.
- **State detachment:** Detach the recurrent hidden state from the autograd graph after each optimizer step, so backprop only spans one timestep (avoid unbounded graph growth / memory blowup from training on an indefinitely long sequence).
- **No offline/external dataset** — training data is exclusively the live camera stream. Frames are either trained on immediately or briefly held in the bounded in-memory replay buffer described above; nothing is ever persisted to disk or survives a process restart.

## Configuration (should be easily editable constants or CLI flags)
- Server host/port, and TLS cert/key paths for browser camera access from off-localhost
- Working resolution (width, height); display upscale factor (also the learned upsample head's fixed integer scale factor -- see Model architecture)
- Upsample head channels/residual-block count/residual-delta-scale (0 channels disables the learned head, falling back to plain bilinear upsampling), and its reconstruction-loss weight
- Learning rate (+ warmup schedule)
- Encoder depth/channel counts, hierarchical ST-LSTM layer count/hidden channels, per-skip-scale recurrent hidden-channel base (0 disables skip recurrence)
- Flow head channels, flow-smoothness weight, flow-acceleration toggle + its own smoothness weight, blend-mask toggle
- Discriminator channels/layers/learning rate, adversarial loss weight + warmup ramp
- Unified multistep self-consistency: pixel weight, latent weight, horizon, cadence (every-N-steps), decay
- Learned-uncertainty-weighting clamp bound
- Motion-magnitude supervision weight, motion-weighted reweighting factor
- Optimizer choice (default Adam)
- Loss spike guard: on/off toggle, rolling-std-multiplier threshold, minimum rolling-history length before it arms (NaN/Inf rejection is unconditional); auto full-reset rejection-rate window length and rejection-count threshold (0 disables auto-reset)
- Real-frame delay in frames (default 0 = every frame is real), adjustable live via a browser slider that snaps to whole-frame detents and shows the current frame-delay value -- this single value doubles as the display forecast's horizon
- Checkpoint file path, autosave interval (0 disables), fresh-start flag to skip loading an existing checkpoint

## Non-goals (explicitly out of scope, don't build)
- Standalone checkpoint MANAGEMENT beyond a single autosave/reload path: no versioned checkpoint history, no browsing/selecting among multiple saved checkpoints, no export/import between machines. Saving/loading model weights itself is now in-scope (see Core behavior / Model architecture) -- this non-goal only bounds its complexity to "one file, autosaved, loaded automatically at startup unless `--fresh-start`."
- Any dataset collection, offline pretraining, or disk-backed training corpus from sources other than the live camera. The replay buffer remains a small, bounded, in-memory FIFO scheduling queue, not a dataset (unchanged from earlier versions: it doesn't persist across restarts, isn't shuffled/sampled from, every frame is consumed in strict arrival order exactly once). The checkpoint file (see above) is the one exception to "nothing survives a restart," and even it persists weights only, never frames.
- Additional auxiliary input modalities beyond the in-scope learned optical-flow head (e.g. depth, segmentation, audio)
- A general-purpose multi-user web service (accounts, auth, session isolation, concurrent independent training runs) -- this remains a single-operator LAN tool. One `TrainingEngine` instance lives for the process lifetime; there's no per-connection isolation.
- Generation as a separate, independently-triggered feature: there is no standalone "Generate" button/API distinct from the delay slider, no saved/downloadable generated clips, no batch/offline generation mode, no generation from an arbitrary (non-live-trained) seed, and no requesting a forecast horizon different from the current delay value. The look-ahead forecast described in Core behavior is strictly a byproduct of the delay slider's existing value, continuously regenerated -- not a separate control surface.

## Deployment / Architecture

The server (`server.py`, FastAPI + Uvicorn) runs on the machine doing the actual training/inference -- typically the one with a GPU. It has no camera of its own. A browser on any device on the same network (e.g. a laptop with a built-in webcam) opens the server's page, captures its own webcam via `getUserMedia`, and streams frames to the server over a WebSocket (`/ws`) as JPEG bytes at display resolution (working resolution x upscale factor) -- the server derives the internal working-resolution tensor from that same frame (a downsize) alongside the full-resolution one (typically no resize at all), so one frame over the wire serves both the model's own input and the upsample head's ground truth. The server streams prediction frames (now genuinely at display resolution, via the learned upsample head) and stats back over the same socket; the "real" pane needs no round trip since the browser already has its own live camera frame locally.

Browsers only allow camera access (`getUserMedia`) over a secure context -- `https://` or `localhost`. Accessing the server from another machine on the LAN therefore requires a TLS certificate; a self-signed one (`--ssl-keyfile`/`--ssl-certfile`, generated with a single `openssl req` command) is sufficient for this use case -- the browser will show a one-time warning to click through.

## Deliverable
A minimal set of Python modules (server-side) plus a small static HTML/JS/CSS frontend, with:
- Dependencies managed via `uv` (`pyproject.toml`/`uv.lock`) -- torch, opencv-python, numpy, fastapi, uvicorn
- Clear inline comments explaining the online-training loop logic (this is the non-obvious part: predict-then-wait-for-ground-truth-then-backprop, one step behind)
- A short usage note at the top of the server entrypoint: how to run it, how to reach it from a browser, and what the on-page controls (delay slider, Reset, Stop) do

## Acceptance criteria
- Opening the served page in a browser and granting camera permission shows two live panes within a few seconds of the WebSocket connecting.
- Both panes update live.
- The prediction pane visibly changes character over the first 30–60 seconds of runtime (starts blurry/noisy, becomes more temporally coherent) — confirming that training is actually happening, not just displaying a static or copied frame.
- Loss value displayed on screen trends downward or stabilizes over time rather than diverging.
- Frame rate is high enough to feel "live" (aim for double-digit FPS on a mid-range GPU; CPU fallback should still run, even if slower), bounded by network round-trip in addition to compute.
- Stop pauses training without crashing or losing hidden state; Reset clears hidden/optimizer state without crashing.
